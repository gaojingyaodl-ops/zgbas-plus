# Quick Task 260718-g93: Fix /index NPE — init auth-sdk ConfigUtil cache on startup

<objective>
Post-login GET `/index` throws NPE at `ConfigUtil.getValue:52` (`configCache.get(appCode)`). Root cause: auth-sdk `ConfigUtil.configCache` is a static Guava `LoadingCache` that stays null until `ConfigUtil.init()` is explicitly called; the monolith startup listener `ApplicationStartup` initializes 11 sibling caches but omits `ConfigUtil.init()`. The old web service had a separate cache-init that called it; that initializer was lost in the web→monolith migration. Fix: add `ConfigUtil.init();` to `ApplicationStartup`.
</objective>

<root_cause_evidence>
- `ConfigUtil.getValue` bytecode line 52 = `getstatic configCache; invokeinterface LoadingCache.get(appCode)` → NPE when configCache null.
- `ConfigUtil.static {}` only initializes the Logger (lines 19, 22-23); `configCache` is built only inside `init()`.
- `ApplicationStartup.onApplicationEvent` (zgbas-system/.../listener/ApplicationStartup.java:40-56) calls `DictUtil.init(appCode)`, `UserCache.init()`, etc. (11 caches) but NOT `ConfigUtil.init()`.
- `IndexController.index:85` calls `ConfigUtil.getValue(...)` unconditionally; line 83 `authOpenFacade.findUserById` already succeeded, so the auth HTTP path is healthy — only the cache init is missing.
- Not a Phase 5 regression (Phase 5 was report-only); a pre-existing startup-wiring gap surfaced during end-to-end verification.
</root_cause_evidence>
