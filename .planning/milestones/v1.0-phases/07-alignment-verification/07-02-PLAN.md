在 `ZgbasApplicationTest.java` 新增以下 @Disabled proof 方法：

**`reportExportSample_proof()`** — @Disabled("D-P7-02 报表导出抽样 proof — 手动启用，需登录 session")

对 5 个报表导出端点做 HTTP 请求（需 Shiro 认证 session，此 proof 需人工启用后先登录）：
1. POST `/rpt/contractReport/exportExcel` — 合同台账导出
2. POST `/rpt/buystatistics/exportExcel` — 采购统计导出
3. POST `/rpt/stat/exportExcel` — 自营审核统计导出
4. POST `/rpt/baseCost/exportExcel` — 业务成本导出
5. POST `/rpt/contractReport/profitexportExcel` — 毛利率导出

断言（每个端点）：
- `response.getStatusCodeValue()` != 404（端点注册）
- 如返回 2xx：断言 `Content-Type` 包含 `xlsx` 或 `octet-stream`，且 body 非空
- 如返回 302/401：端点可达但需认证（可接受，证明路由正确）

注：因为 TestRestTemplate 无 Shiro session，多数导出端点可能返回 302/401。此 proof 主要验证端点非 404 + Content-Type 正确性（在认证通过时）。人工启用时可搭配浏览器 session 验证完整导出。

**`reportFindPageSample_proof()`** — @Disabled("D-P7-02 报表分页抽样 proof — 手动启用")

对 3 个报表 findPage 端点做 Mapper 级查询（不走 HTTP，直接调 Mapper，沿用 sampleReportQuery_proof 模式）：
1. `rptFundReceivableStatisticsMapper.findPage(vo)` — 断言返回非空 + size ≤ vo.getRows()
2. `rptCtrContractReportMapper.findRptContractPage(vo)` — 断言返回非空
3. `rptBusinessOverviewMapper.findBusinessOverviewList(vo)` — 断言返回非空

额外分页正确性断言：
- 设置 page=1, rows=5 → 断言结果 size ≤ 5
- 设置 page=1, rows=50 → 断言结果 size ≥ 5（如 DB 有足够数据）
- 这验证 D-P5-06 "1:1 等价" 分页正确性

需要注入的 Mapper 已在 ZgbasApplicationTest 中存在（Phase 5 注入）。
</action>
